## 产品摘要

- 为已发布的公开文章补充阅读量统计能力，文章详情页在内容成功渲染后会触发一次计数，且不影响标题、摘要、正文和元信息展示。
- 后端新增公开阅读记录接口 `POST /api/articles/:slug/view`，并保持 `GET /api/articles/:slug` 为纯读取。
- `Article` / `ArticleSummary` / `ArticleDetail` 统一增加 `viewCount` 字段，API 响应和 SDK 解析结果同步更新。
- 数据库为 `articles` 表新增 `view_count INTEGER NOT NULL DEFAULT 0 CHECK (view_count >= 0)`，历史文章自动获得默认值 0。
- 后台文章列表或 Dashboard 至少一处展示阅读量，便于管理者查看内容表现。

## 用户影响

- 访客阅读已发布文章时，页面会展示并累积阅读量，但不会改变原有浏览体验。
- 草稿和未发布文章不计入公开阅读量，避免污染统计口径。
- 管理端可以直接查看每篇文章的阅读表现，便于内容运营和复盘。

## 需求摘要

## 背景

当前项目是个人博客/内容管理系统，已具备公开文章列表、文章详情、分类/标签、后台文章管理和基础 Dashboard 能力。现有“阅读时间”主要用于估算文章阅读耗时，不等同于真实阅读量统计；文章数据模型、公开页面和后台运营视图中尚未稳定呈现“文章被阅读次数”。

本轮迭代需要增加阅读量统计能力：当公开端已发布文章详情页被成功阅读后，系统记录该文章阅读次数，并在 C 端和后台管理端展示，帮助读者感知内容热度，也帮助管理员判断文章表现。

用户已确认本轮默认口径为：公开文章详情页成功阅读后计数，C 端展示阅读量，后台文章列表或 Dashboard 展示阅读量，并包含必要的数据字段和数据库迁移评估。

## 目标

- 为已发布公开文章建立阅读量统计能力，文章详情页成功阅读后该文章阅读量按统一口径增加。
- 在 C 端公开文章详情页展示阅读量，展示位置应服务阅读体验，不干扰标题、摘要、正文和元信息阅读。
- 在后台文章列表或 Dashboard 至少一处展示文章阅读量，供管理员/编辑查看内容表现。
- 在 API 响应、共享类型、SDK、web 前端和 admin 前端之间保持阅读量字段语义一致，建议字段语义为文章维度总阅读次数。
- 将数据字段和数据库迁移纳入本轮评估范围，确保现有文章具备默认阅读量，且后续可持久保存统计结果。
- 明确边界：不存在、草稿、未发布文章不产生公开阅读量；统计失败不阻塞公开文章详情阅读。

## 非目标

- 不建设完整数据分析平台，不包含趋势图、来源渠道、用户画像、停留时长、转化漏斗或历史分时统计。
- 不做唯一访客识别、登录用户行为追踪、跨设备去重、IP/设备指纹采集或复杂反作弊策略。
- 不引入第三方统计平台、埋点 SDK 或外部分析服务。
- 不提供管理员手动修改、重置或校准阅读量的后台操作入口。
- 不新增独立“阅读量管理”页面；后台只在现有文章列表、Dashboard 或文章相关视图中呈现。
- 不要求公开列表页、首页、分类页、标签页必须展示阅读量；如实现方选择展示，需与详情页字段来源一致。

- [ ] AC-2：访问不存在文章、草稿文章、未发布文章或未成功返回文章内容的详情请求，不增加公开阅读量。

## 回滚方式

- 若发布后阅读量展示异常但文章核心阅读正常，优先回滚或关闭 C 端和后台阅读量展示入口，保留文章详情、列表和后台管理原有流程可用。
- 若统计写入导致文章详情访问失败、明显变慢或产生错误，应回滚统计触发逻辑，确保公开文章详情读取恢复为只读路径。
- 若数据库迁移已执行且需要回滚，应按迁移方案处理新增字段或统计表；回滚前需确认是否保留已产生的阅读量数据，避免误删有价值统计。
- 若共享类型、SDK 或前端字段兼容出现问题，应同步回滚相关 schema/类型/调用链变更，避免 API 与前端字段契约不一致。
- 回滚后文章阅读量可以暂时不展示或展示为安全缺省状态，但不得影响文章正文访问、后台登录和文章管理主流程。
