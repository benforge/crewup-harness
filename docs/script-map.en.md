This document defines CrewUp's script boundaries for maintainers. The goal is not to make users learn every script, but to keep a stable product surface and avoid growing the harness through scattered patches.

## Core Product Entry Points

These commands should remain stable, documented, and recommended to regular developers:

| Command | Script | Role |
| --- | --- | --- |
| `crewup install` | `bin/crewup.mjs` | Install or upgrade the harness template |
| `crewup doctor` | `doctor.mjs` | Check local environment and available capabilities |
| `crewup inspect --no-ai` | `inspect.mjs` | Optional scan for existing or complex projects |
| `crewup init` | `init.mjs` | Generate project adaptation |
| `crewup check` | `check.mjs` | Validate harness integrity |
| `crewup run` | `run.mjs` | Formal run entry |
| `crewup next-agent` | `next-agent.mjs` | Show currently runnable subagents |
| `crewup native-state` | `native-state.mjs` | Register and diagnose native subagent state/results |
| `crewup audit` | `orchestration-audit.mjs` | Audit orchestration order, owner boundaries, repair loops, and context pressure |
| `crewup gate-check` | `gate-check.mjs` | Quality gates, ownership, and overreach checks |
| `crewup report` | `report.mjs` | Delivery report |
| `crewup finish` | `finish.mjs` | Done/archive closeout |

## Role Source Of Truth

Agent role sets and execution order are centralized in `.harness/scripts/lib/agent-roles.mjs`. Dispatch, gates, native-state, native-plan, and transitions should import that contract instead of maintaining separate role lists.

| Group | Agents |
| --- | --- |
| Planning | `pm`, `requirements-plan`, `requirements`, `architect` |
| Implementation | `frontend`, `docs`, `backend`, `database`, `devops` |
| Code implementation | `frontend`, `backend`, `database`, `devops` |
| Write owner | `frontend`, `docs`, `backend`, `database`, `devops`, `tester` |
| Verification/release | `tester`, `reviewer`, `release` |

## Consolidation Direction

- `finalize.mjs` has been removed; use `finish.mjs`.
- `requirements-interview.mjs` and `requirements-plan.mjs` have been removed; requirement planning artifacts must be written by the `requirements-plan` / `requirements` subagents.
- `desktop-plan.mjs` and `desktop-light.mjs` have been removed; non-native environments use the bridge handoff generated by `native-plan.mjs`.
- `orchestrate.mjs` is advanced bridge-path support. The Codex-native path should prefer `native-plan`, `next-agent`, and `native-state`.
- `next.mjs` is a status advisor, not an executor. Formal dispatch should use `next-agent`.

## Current Core Workflow Contract

1. `crewup run` may generate candidate agents from the entry request, but implementation agents are candidates only.
2. `requirements-plan`, `requirements`, and `architect` must complete in order.
3. `architect` owns `implementation-plan.md` and must assign implementation work using exact agent ids.
4. `next-agent` and `native-state` use `implementation-plan.md` to decide which implementation agents may actually start.
5. Implementation candidates not assigned by `implementation-plan.md` are skipped and do not block tester.
6. Tester/reviewer feedback must route back to owner implementation agents; the main agent must not directly fix business code.
7. `gate-check` audits owner artifacts, native results, changed files, review/test state, and overreach risk.
