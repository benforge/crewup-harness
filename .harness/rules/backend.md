# 后端规则

## 基本原则

- 先读项目已有代码和规范，再写代码。
- 优先复用项目内已有模块、类型、工具函数、错误处理和配置方式。
- 不为了单个需求引入大型框架或基础设施，除非方案中已经说明并获得确认。
- 后端实现要保证接口契约清晰、边界可测试、失败可解释。
- 涉及安全、权限、数据一致性、生产部署时，必须保守处理并记录风险。

## 实现前检查

- 当前需求影响哪些服务模块或 API。
- 是否需要新增、修改或废弃接口。
- 是否需要数据库模型、迁移或索引配合。
- 是否影响认证、授权、角色、租户或访问边界。
- 是否需要事务、幂等、限流、审计、日志或异步任务。
- 是否需要更新共享类型、SDK、OpenAPI 或前端调用方式。

## 编码规则

- 接口输入必须有明确校验，不能信任客户端传入。
- 错误处理要稳定，避免把内部异常直接暴露给调用方。
- 领域逻辑不要散落在路由或控制器里，按项目既有分层组织。
- 数据访问要考虑事务、一致性、索引和并发风险。
- 外部服务调用要考虑超时、重试、降级和错误记录。
- 不把测试数据、密钥、Token、真实账号写进代码。

## 契约规则

- 新增或修改 API 时，必须更新 `artifacts/api-change.md`。
- 涉及数据库时，必须通知 Database Agent 或更新 `artifacts/db-migration.md`。
- 涉及部署、环境变量或云服务时，必须通知 DevOps Agent 或更新发布说明。
- 涉及前端调用时，必须同步共享类型或 SDK 变化。

## 验证规则

- 至少记录本次后端变更的接口验证路径。
- 有测试框架时，补充与风险匹配的单元测试或集成测试。
- 涉及权限、安全、数据写入时，必须记录正向和反向验证。
- 无法验证时，写明原因和剩余风险。
