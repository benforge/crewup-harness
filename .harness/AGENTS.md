# Harness Agent Rules

These rules define the reusable CrewUp workflow layer. They are project-agnostic by default and do not assume a specific app directory, framework, language, deployment target, or business domain.

## Core Model

- `.harness/config/` defines workflow, delegation, risk, model, document, service, and quality policies.
- `.harness/orchestrator/` defines how the main agent routes work and coordinates subagents.
- `.harness/agents/` and `.harness/rules/` define role behavior.
- `.harness/templates/` defines artifact templates.
- `.harness/scripts/` provides CLI commands for install, inspect, init, run creation, state transitions, reports, checks, knowledge refresh, service lifecycle, and archive commits.
- `.harness/project/` is the target-project adaptation layer generated by `crewup init`.
- `.harness/runs/`, `.harness/reports/`, `.harness/dashboard/`, and `.harness/knowledge/` are runtime/state directories.

## Activation

- CrewUp does not take over every chat.
- Enter the formal harness workflow only when the user runs `crewup run` / `npm run harness:run`, or explicitly asks to use CrewUp, follow the harness workflow, or continue an existing CrewUp run.
- Without an explicit CrewUp signal, simple Q&A, read-only explanation, temporary discussion, and tiny informal fixes stay outside the harness and do not create a run.
- Once a run exists, formal project work must follow delegation, artifact ownership, quality gates, reports, service lifecycle, and archive rules.
- CrewUp is intended for large or formal engineering work. Small work that is not worth delegation should stay outside the harness instead of turning the main agent into an implementer.

## Main Agent Boundary

- The main agent coordinates work, creates/selects runs, prepares tasks, starts subagents, captures results, checks gates, and summarizes outcomes.
- The main agent must not author formal owner artifacts or business code when the responsible subagent exists.
- The main agent may update orchestration metadata, state, task handoffs, reports, archive records, and other runtime files owned by coordination.
- Tester/reviewer feedback that requires code changes must be delegated back to implementation owner agents.
- If native subagents are unavailable, record fallback and stop at coordination/reporting; fallback does not authorize the main agent to complete delegated work.

## Encoding

- Harness files are UTF-8.
- When inspecting local text through a shell, use explicit UTF-8 handling, such as `Get-Content <file> -Encoding UTF8` or a Node UTF-8 read.
- Do not judge Chinese documentation from mojibake terminal output. Verify with UTF-8 first.
- Keep machine-checked contracts in English: artifact headings, JSON fields, command names, status values, and schema labels.

## Required Owner Boundaries

- `requirements-plan` owns `artifacts/requirement-plan.md`.
- `requirements` owns `artifacts/requirement.md`.
- `architect` owns `artifacts/architecture.md` and `artifacts/implementation-plan.md`.
- `frontend`, `backend`, `database`, `devops`, and `docs` own implementation or documentation files inside their allowed write scopes.
- `tester` owns `artifacts/test-report.md`.
- `reviewer` owns `artifacts/review-report.md`.
- `release` owns `artifacts/release-summary.md`.

## Strict Workflow

- Explicit full-loop or strict workflow requests stay on the full workflow.
- Full workflow may include PM, requirements-plan, requirements, architect, implementation agents, tester, reviewer, and release.
- Reducing token cost must come from clearer contracts, narrower context, better generated tasks, and fewer false-positive gates, not from bypassing required roles.

## Project Adaptation

Every target project should generate its own adaptation layer:

```bash
npm run harness:inspect -- --no-ai
npm run harness:init
npm run harness:check
```

Use `--ai` only when API access is configured and the user wants model-assisted adaptation from real project evidence.

## Completion

A formal run is closed only when required gates pass or blockers are recorded:

- owned artifacts exist and match schema
- native/bridge results are captured
- changed files are recorded and match owner scopes
- tests and review are recorded
- release summary exists
- preview/dev services are stopped before done/archive
- close_required subagents are closed
- archive policy has run or recorded a skip/failure reason
