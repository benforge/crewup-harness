# 需求扩写草案

> 这是 plan-only 阶段产物。用户确认后，才可以提升为正式 `requirement.md` 并进入架构/实现流程。

## 原始需求摘要

- 概述用户原始诉求，保留关键对象、范围和约束。

## 需求背景

- 说明为什么需要这次迭代；如果背景未知，写“用户未补充额外业务背景”。

## 过往背景

- 相关历史 run：阅读 `logs/context/related-runs.md` 后填写；未命中则写“未命中相关历史 run”。
- 可复用背景：无。
- 冲突或变化：无。

## 目标

- 写成可执行、可验收的目标列表。

## 非目标

- 写清本轮不做什么；如果没有额外限制，写“无额外非目标”。

## 用户角色与权限

- 写明受影响角色；如果不涉及权限变化，写“不涉及权限变化”。

## 核心用户流程

- 写明主要操作路径；如果是纯技术改造，写“用户流程保持不变”。

## 页面与路由候选

- 写明候选页面/路由；如果不涉及页面，写“不涉及页面或路由变化”。

## 数据与接口线索

- 写明相关 API、数据模型或环境变量；如果不涉及，写“不涉及接口或数据变化”。

## 验收标准

- [ ] AC-1：补充可检查的验收标准。

## 影响范围候选

- [ ] web
- [ ] admin
- [ ] api
- [ ] db
- [ ] infra
- [ ] docs

## 风险与边界

- 写明实现边界、潜在回归点和需要人工确认的风险；如无明显风险，写“无高风险项”。

## 待确认问题

- 无。
