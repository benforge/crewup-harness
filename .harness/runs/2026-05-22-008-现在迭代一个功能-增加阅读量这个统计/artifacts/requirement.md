# 阅读量统计需求文档

## 背景

当前项目是个人博客/内容管理系统，已具备公开文章列表、文章详情、分类/标签、后台文章管理和基础 Dashboard 能力。现有“阅读时间”主要用于估算文章阅读耗时，不等同于真实阅读量统计；文章数据模型、公开页面和后台运营视图中尚未稳定呈现“文章被阅读次数”。

本轮迭代需要增加阅读量统计能力：当公开端已发布文章详情页被成功阅读后，系统记录该文章阅读次数，并在 C 端和后台管理端展示，帮助读者感知内容热度，也帮助管理员判断文章表现。

用户已确认本轮默认口径为：公开文章详情页成功阅读后计数，C 端展示阅读量，后台文章列表或 Dashboard 展示阅读量，并包含必要的数据字段和数据库迁移评估。

## 目标

- 为已发布公开文章建立阅读量统计能力，文章详情页成功阅读后该文章阅读量按统一口径增加。
- 在 C 端公开文章详情页展示阅读量，展示位置应服务阅读体验，不干扰标题、摘要、正文和元信息阅读。
- 在后台文章列表或 Dashboard 至少一处展示文章阅读量，供管理员/编辑查看内容表现。
- 在 API 响应、共享类型、SDK、web 前端和 admin 前端之间保持阅读量字段语义一致，建议字段语义为文章维度总阅读次数。
- 将数据字段和数据库迁移纳入本轮评估范围，确保现有文章具备默认阅读量，且后续可持久保存统计结果。
- 明确边界：不存在、草稿、未发布文章不产生公开阅读量；统计失败不阻塞公开文章详情阅读。

## 非目标

- 不建设完整数据分析平台，不包含趋势图、来源渠道、用户画像、停留时长、转化漏斗或历史分时统计。
- 不做唯一访客识别、登录用户行为追踪、跨设备去重、IP/设备指纹采集或复杂反作弊策略。
- 不引入第三方统计平台、埋点 SDK 或外部分析服务。
- 不提供管理员手动修改、重置或校准阅读量的后台操作入口。
- 不新增独立“阅读量管理”页面；后台只在现有文章列表、Dashboard 或文章相关视图中呈现。
- 不要求公开列表页、首页、分类页、标签页必须展示阅读量；如实现方选择展示，需与详情页字段来源一致。

## 验收标准

- [ ] AC-1：访问已发布公开文章详情页并成功返回/渲染文章内容后，该文章阅读量增加 1；连续多次成功访问按本轮默认口径分别计数。
- [ ] AC-2：访问不存在文章、草稿文章、未发布文章或未成功返回文章内容的详情请求，不增加公开阅读量。
- [ ] AC-3：公开文章详情页展示阅读量，文案清晰，例如“123 次阅读”或等价表达；阅读量为 0、缺失或接口降级时不出现 `undefined`、`NaN`、空白错位或误导性数值。
- [ ] AC-4：后台文章列表或 Dashboard 至少一处展示文章阅读量，且该信息只对已登录并具备后台访问权限的管理员/编辑可见。
- [ ] AC-5：阅读量字段在公开文章详情、后台文章数据、共享类型、SDK 解析和前端调用链路中保持一致；构建、类型检查或等价校验不因字段缺失失败。
- [ ] AC-6：现有文章在新增字段或迁移后具备明确默认值，默认阅读量为非负整数，历史文章不会因缺少阅读量字段导致 API 或页面报错。
- [ ] AC-7：统计写入失败时，公开文章详情页仍能正常展示正文；系统应保留可诊断错误信息或降级路径，不能把统计失败暴露为读者不可阅读。
- [ ] AC-8：移动端与桌面端文章元信息区域布局稳定，阅读量展示不会挤压标题、摘要、作者、发布时间、分类/标签和阅读时间。
- [ ] AC-9：如果公开文章列表、首页、分类页或标签页展示阅读量，展示数值必须与文章详情同一字段来源一致；如果不展示，原有布局和跳转不受影响。
- [ ] AC-10：本轮交付包含数据字段/迁移评估结论；若实施数据库迁移，应说明默认值、兼容旧数据和回滚路径。

## 影响范围

- [x] web：公开文章详情页需展示阅读量；如复用文章卡片或列表数据，需要保证缺省展示策略稳定。
- [x] admin：后台文章列表或 Dashboard 至少一处展示阅读量，并遵守现有登录态和后台权限边界。
- [x] api：公开文章详情读取/成功阅读计数、后台数据返回、文章摘要/详情类型需要包含一致的阅读量字段。
- [x] db：需要评估并可能新增文章阅读量字段或等价持久化结构，覆盖默认值、迁移、兼容旧数据和回滚。
- [x] docs：需求、实现说明、测试报告或发布摘要中需记录统计口径、字段含义和回滚方式。
- [ ] infra：本轮默认不新增基础设施；如后续架构评估发现需要缓存、队列或限流，再单独确认。

## 测试要求

- API 测试需覆盖：已发布文章详情成功访问后阅读量递增；不存在、草稿、未发布文章不计数；返回数据包含非负整数阅读量字段。
- 后台测试或验证需覆盖：管理员/编辑可在文章列表或 Dashboard 查看阅读量；未授权访问者不能获取后台阅读量管理信息。
- 前端测试或人工验证需覆盖：公开文章详情页阅读量文案正常；0 值、缺失值、接口降级时展示稳定；移动端和桌面端布局无明显挤压或错位。
- 数据/迁移测试需覆盖：现有文章迁移后默认阅读量可用；旧数据缺少字段时 API、SDK、web、admin 均不报错。
- 回归验证需覆盖：文章列表、分类页、标签页、首页推荐、文章详情、后台文章管理的原有核心流程不因新增阅读量受到破坏。
- 如统计写入被模拟为失败，需验证文章详情仍可阅读，并能在日志、错误返回或测试断言中观察到可诊断信息。

## 回滚方式

- 若发布后阅读量展示异常但文章核心阅读正常，优先回滚或关闭 C 端和后台阅读量展示入口，保留文章详情、列表和后台管理原有流程可用。
- 若统计写入导致文章详情访问失败、明显变慢或产生错误，应回滚统计触发逻辑，确保公开文章详情读取恢复为只读路径。
- 若数据库迁移已执行且需要回滚，应按迁移方案处理新增字段或统计表；回滚前需确认是否保留已产生的阅读量数据，避免误删有价值统计。
- 若共享类型、SDK 或前端字段兼容出现问题，应同步回滚相关 schema/类型/调用链变更，避免 API 与前端字段契约不一致。
- 回滚后文章阅读量可以暂时不展示或展示为安全缺省状态，但不得影响文章正文访问、后台登录和文章管理主流程。
