# 需求扩写草案

> 这是 plan-only 阶段产物。用户确认后，才可以提升为正式 `requirement.md` 并进入架构/实现流程。

## 原始需求摘要

- 为个人博客/内容管理系统增加“阅读量”统计能力，让文章被阅读后的浏览数据可以被记录、展示和用于后台运营判断。

## 需求背景

- 当前公开博客已经包含文章列表、文章详情、分类/标签、后台文章管理和基础 Dashboard，但文章数据模型与页面展示中尚未体现阅读量。
- 用户希望本轮迭代补齐阅读统计能力，帮助判断文章受欢迎程度，并在合适位置向读者或管理员展示阅读情况。
- 该需求处于 plan-only 阶段，具体统计口径、存储方式和防刷策略仍需后续确认或由架构阶段评估。

## 目标

- 为已发布文章建立阅读量统计口径，至少能统计公开文章详情页被访问/阅读的次数。
- 在公开端文章信息中展示阅读量，优先不打断正文阅读体验。
- 在后台文章管理或 Dashboard 中展示阅读量，帮助管理员判断内容表现。
- 确保阅读量字段在 API、类型定义、SDK/前端调用链路中保持一致，不出现字段漂移。
- 统计行为对草稿、未发布文章和不存在的文章保持清晰边界。

## 非目标

- 不在本阶段建设完整数据分析平台、趋势图、用户画像、来源渠道、停留时长或转化漏斗。
- 不要求识别唯一真实用户、登录用户行为或跨设备去重，除非用户后续明确要求。
- 不默认引入第三方统计平台、埋点 SDK 或复杂反作弊系统。
- 不把后台管理控件暴露到 C 端公开阅读页面。
- 不在 plan-only 阶段决定数据库结构、缓存策略、接口方法或具体实现方案。

## 用户角色与权限

- 普通访问者：
  - 可访问公开文章列表、分类/标签页和文章详情页。
  - 访问已发布文章详情时可能触发阅读量统计。
  - 可在公开页面看到被允许展示的阅读量信息，不需要登录。
- 管理员/编辑：
  - 登录后台后可在文章列表、文章详情编辑页或 Dashboard 查看阅读量。
  - 可用阅读量辅助内容运营判断，但默认不能手动篡改阅读量，除非后续确认需要校准入口。
- 未授权访问者：
  - 不能访问后台阅读量管理信息。
  - 对草稿或未发布文章的访问不应产生公开阅读量。

## 核心用户流程

- 公开阅读流程：
  1. 访问者进入文章列表、首页推荐、分类页或标签页。
  2. 访问者点击进入 `/articles/[slug]` 文章详情。
  3. 系统判断文章存在且已发布后，按确认后的统计口径记录一次阅读。
  4. 文章详情页在元信息区域展示阅读量，与发布时间、作者、阅读时间等信息保持同级或次级关系。
- 后台查看流程：
  1. 管理员登录后台。
  2. 进入 Dashboard 或 Articles 列表。
  3. 查看文章阅读量，可按现有列表信息辅助判断内容表现。
  4. 若接口失败，后台应保留现有错误提示/降级体验，不展示误导性统计。
- 异常流程：
  1. 访问不存在、未发布或草稿文章时，不记录公开阅读量。
  2. 统计写入失败时，公开阅读页面仍应可正常渲染文章内容。
  3. fallback 本地文章数据是否展示阅读量需明确；默认可展示占位或隐藏，不应伪造线上统计。

## 页面与路由候选

- C 端公开页面候选：
  - `/articles/[slug]`：文章详情页，最核心的阅读量展示与统计触发位置。
  - `/articles`：文章列表页，可选展示每篇文章阅读量，用于内容扫描。
  - `/categories/[slug]`、`/tags/[slug]`：分类/标签文章列表，可选展示阅读量，保持与文章列表一致。
  - `/`：首页推荐文章区域如已有文章卡片，可选同步展示阅读量。
- Admin 后台页面候选：
  - `/dashboard`：可选增加总阅读量、最高阅读文章或近期内容表现摘要。
  - `/articles`：文章库存列表中增加阅读量列，支持管理员快速比较文章表现。
  - `/articles/[id]`：文章编辑页侧栏可选展示当前阅读量，只读展示。
- 不建议新增独立“阅读量管理”页面，除非后续确认需要趋势、重置、人工校准或审计能力。

## 数据与接口线索

- 现有 API 线索：
  - 公开文章列表：`GET /api/articles`
  - 公开文章详情：`GET /api/articles/:slug`
  - 后台文章列表：`GET /api/admin/articles`
  - 后台文章详情：`GET /api/admin/articles/:id`
- 候选数据字段：
  - `viewCount` 或同等语义字段，类型为非负整数，出现在文章摘要和文章详情中。
  - 可选 `lastViewedAt`、`uniqueViewCount`、`viewedAt` 等扩展字段暂不纳入最小目标，除非用户确认。
- 候选统计接口/行为：
  - 文章详情读取时自动计数，或由前端在详情页加载成功后调用独立统计接口。
  - 若采用独立接口，需评估幂等、限流、错误处理和是否允许匿名请求。
- 类型与调用链路候选：
  - `apps/api/src/articles/types/article.types.ts`
  - `packages/types/src/index.js`
  - `packages/sdk/src/index.js`
  - `apps/web/lib/api.ts`
  - `apps/admin/src/main.tsx`
- 存储候选：
  - 当前存在 in-memory repository 与数据库迁移目录；最终存储方式需由架构阶段基于现有持久化策略确认。

## 验收标准

- [ ] AC-1：访问已发布文章详情页后，该文章的阅读量会按确认口径增加；访问不存在、未发布或草稿文章不增加公开阅读量。
- [ ] AC-2：公开文章详情页展示阅读量，文案清晰，例如“123 次阅读”或等价表达；阅读量为 0、缺失或接口降级时有明确展示策略，不出现 `undefined`、`NaN` 或空白错位。
- [ ] AC-3：后台文章列表或 Dashboard 至少一处展示每篇文章阅读量，且只有已登录管理员/编辑可见。
- [ ] AC-4：公开文章列表、分类页、标签页或首页如展示阅读量，应与文章详情字段来源一致；如不展示，需要保持页面布局不受影响。
- [ ] AC-5：阅读量字段在 API 响应、共享类型、SDK 和前端类型中保持一致，前后端构建/类型检查通过。
- [ ] AC-6：统计写入失败不阻塞文章详情阅读；用户仍能看到正文，系统可记录或返回可诊断错误。
- [ ] AC-7：移动端与桌面端文章元信息区域布局稳定，阅读量不会挤压标题、摘要、作者、发布时间、分类/标签和阅读时间。
- [ ] AC-8：测试或验证覆盖正向计数、草稿/未发布不计数、后台展示、接口失败降级和字段兼容。

## 风险与边界

- 统计口径风险：刷新页面、重复访问、搜索引擎爬虫、预渲染/构建访问、接口重试是否计数，需要明确，否则数据可信度会波动。
- SEO/静态渲染边界：公开文章详情当前存在静态参数与服务端渲染逻辑，不能让构建期或元信息生成误触发阅读量。
- 数据一致性风险：阅读量字段需要在 API、共享类型、SDK、web 和 admin 中同步，否则容易出现展示字段缺失或运行时报错。
- 性能与写入压力：热门文章可能导致频繁写入，后续架构需评估是否需要限流、缓存、批量更新或异步统计。
- 隐私边界：最小需求只统计文章维度阅读次数，不采集 IP、设备指纹、用户标识等个人数据；若后续需要去重，应先确认隐私与合规边界。
- 迁移风险：如果落到数据库字段或统计表，需要考虑现有文章默认值、回滚路径和旧数据兼容。
- fallback 数据边界：本地 fallback 文章不代表线上统计，不应写入真实阅读量，也不应展示误导性数值。

## 待确认问题

- 阅读量统计口径：每次文章详情请求都算一次，还是同一浏览器/同一时间窗口内去重？
- 是否需要排除管理员后台预览、爬虫、构建期请求或本机开发请求？
- 阅读量是否只展示在文章详情，还是列表页、首页、分类/标签页也要展示？
- 后台最小展示位置是 Articles 列表、Dashboard，还是二者都需要？
- 是否允许管理员手动校准阅读量，或仅允许系统自动累计？
- 阅读量是否需要历史趋势、排行、最近 7/30 天统计，还是本轮只做总阅读量？
- 统计数据是否需要持久化到数据库；若是，是否接受本轮包含数据库迁移？
- 阅读量初始值如何处理：现有文章统一为 0，还是允许导入/配置历史值？
- 对统计失败是否需要后台告警/日志，还是仅保证不影响公开阅读？
