# 需求说明

## 背景

本 run 是“照片墙 + COS 存储规划 + Markdown 阅读体验 + 首页简化”的需求细化阶段。当前阶段只输出可开发、可验收的需求与边界，不进入开发实现，不修改 `apps/`，不写入 `docs/product/`。

当前博客 MVP 已具备公开文章、标签、分类与后台内容管理基础，但下一阶段需要让站点更符合“IT 工程师个人技术博客”的表达重点：

- 首页信息需要收敛，只突出个人介绍、文章和标签，避免把照片墙、分类、统计、装饰区块等内容放到同等优先级。
- 新增照片墙能力，用瀑布流展示项目现场、工作台、生活记录或技术活动图片，增强个人站真实感，但不抢首页主线。
- 图片资源未来计划对接腾讯云 COS，因此需求层面必须明确密钥、签名 URL、临时访问、失败降级等边界。
- 文章详情需要支持更完整的 Markdown/GFM 阅读体验，尤其是代码块、高亮、目录、图片、链接和安全 HTML 策略。

## 目标

- 明确首页信息架构：首屏和主体内容只围绕个人介绍、文章、标签展开；照片墙只作为轻量入口存在。
- 明确照片墙 C 端体验：规划 `/photos`、瀑布流、筛选、详情、加载态、空态、错误态、移动端、性能和无障碍要求。
- 明确后台照片管理需求：支持照片元数据、展示状态、排序、上传/登记流程、成功失败提示和权限边界。
- 明确 COS 未来对接约束：服务端持有密钥，前端不保存长期凭证；支持签名 URL、临时/过期访问和失败降级。
- 明确文章详情 Markdown 能力：支持 GFM、代码块、代码高亮、目录、图片、链接，并默认采用安全 HTML 策略。
- 输出可测试、可截图验收的标准，供后续 architect、frontend、backend、database、tester、reviewer agent 使用。

## 非目标

- 本阶段不开发、不修改 `apps/` 下任何业务代码。
- 本阶段不新增真实 COS 密钥、环境变量、生产配置、CI/CD 配置或云资源。
- 本阶段不执行数据库迁移，不上传真实图片，不接入真实 COS 桶。
- 本阶段不写入 `docs/product/`，后续仅在用户确认或 release 同步后沉淀。
- 首期不要求批量上传、批量编辑、EXIF 自动读取、地点地图、相册分组、图片处理流水线或 CDN 深度优化。
- 首期不要求 Markdown 自定义扩展语法、在线协同编辑或所见即所得编辑器。

## 用户角色与权限

- 公开访问者：无需登录，可访问首页、文章列表/详情、标签页、照片墙公开列表和公开照片详情。
- 内容管理员：登录后台后可创建、编辑、隐藏、排序和删除照片记录，可登记图片 URL 或触发未来上传流程。
- 未授权用户：不可访问后台照片管理、上传签名、媒体登记等管理能力；访问时应被拦截到登录或无权限提示。
- 系统服务端：负责持有 COS 密钥、生成签名 URL、处理上传登记、校验文件元数据和返回可展示资源地址。

## 核心用户流程

- 首页浏览流程：访问者进入 `/`，先看到个人介绍，再浏览精选/最新文章，随后通过标签进入聚合内容；照片墙只通过导航或轻量文字/图标入口进入。
- 照片墙浏览流程：访问者进入 `/photos`，看到瀑布流列表；可按标签/分类/年份筛选；点击照片进入详情或打开轻量详情层；加载失败时可重试或返回列表。
- 照片管理流程：管理员进入后台照片管理，新增照片记录，填写标题、描述、标签、拍摄时间、展示状态、排序值和图片来源；保存成功后得到明确提示，失败时看到字段级或全局错误。
- 图片访问流程：公开页面请求照片数据时，只拿到可展示 URL 或签名 URL，不接触 COS 密钥；URL 过期或拉取失败时展示占位图和可理解错误提示。
- 文章阅读流程：读者打开文章详情，页面渲染 Markdown/GFM 内容、目录、图片、链接和代码高亮；代码高亮失败时降级为可读纯文本代码块。

## 页面与路由清单

- 首页 `/`：核心层级为个人介绍、文章、标签。照片墙入口可放在主导航、页尾或文章/标签后的轻量链接，不展示大面积照片预览。
- 文章列表 `/articles`：保持文章浏览入口，不因照片墙需求改变主流程。
- 文章详情 `/articles/[slug]`：增强 Markdown/GFM、代码块、高亮、目录、图片和链接体验。
- 标签聚合 `/tags/[slug]`：作为首页标签入口后的内容聚合页。
- 照片墙 `/photos`：新增公开页面，提供瀑布流、筛选、加载/空/错状态和移动端布局。
- 照片详情 `/photos/[id]` 或列表内详情层：首期可由 architect 决定最终形态，但需求要求桌面端和移动端都能查看大图、标题、描述、标签、拍摄时间和返回路径。
- 后台照片管理页：具体后台路由由现有后台结构决定，需求要求它是受保护管理入口，包含列表、筛选、创建、编辑、状态切换和排序管理。

## 首页信息架构

- 第一优先级：个人介绍。应说明站主身份、技术方向和个人站定位，首屏内可直接理解“这是谁、写什么、为什么值得读”。
- 第二优先级：文章。展示最新或精选文章，包含标题、摘要、发布时间、标签入口，保证访问者能快速进入阅读。
- 第三优先级：标签。展示核心标签或标签云，用于表达内容版图并引导聚合页。
- 照片墙入口：只做轻量处理，不进入首页核心内容层级；可作为主导航项、页尾链接或小型“Photos”入口，不放大面积瀑布流、不在首屏抢视觉焦点。
- 不接受的首页形态：照片墙成为首屏主体；标签、文章、照片、分类、统计等模块均分权重；使用大量装饰性卡片导致个人介绍和文章入口不清晰。

## 照片墙需求

- `/photos` 应以瀑布流为主展示形态，支持不同图片比例，图片卡片高度不强行裁成同一比例。
- 卡片基础信息至少包含图片、标题或简短说明；可选展示标签、拍摄时间或分类，但不得让列表过度拥挤。
- 筛选至少预留标签、分类、年份三类维度；首期若实现筛选，筛选结果必须能被清空并恢复全部列表。
- 详情应展示大图、标题、描述、标签、拍摄时间、来源/项目说明和返回/关闭操作。
- 加载态应有骨架屏或明确加载提示，不能出现页面空白。
- 空态应说明当前没有照片或筛选无结果，并提供清空筛选/返回全部照片的操作。
- 错误态应说明照片加载失败或资源暂不可用，并提供重试入口；单张图片失败时应显示占位图，不影响其他图片继续浏览。
- 移动端应采用单列或适合小屏的瀑布流布局，筛选入口收敛为抽屉、弹层或紧凑控件，详情查看不应横向溢出。
- 性能要求：列表首屏优先加载可见图片；图片应具备懒加载、尺寸约束和占位尺寸，减少布局跳动；大图详情不阻塞列表首屏渲染。
- 无障碍要求：图片必须有可理解的 `alt` 文案来源规则；筛选、详情打开/关闭、重试按钮可键盘操作；详情层若采用弹层，需要焦点管理和关闭方式。

## 后台管理需求

- 照片列表：支持查看缩略图、标题、展示状态、排序、标签/分类、拍摄时间、创建/更新时间。
- 元数据字段：标题、描述、图片地址或对象 Key、缩略图地址或对象 Key、标签、分类、拍摄时间、展示状态、排序值、封面/精选标记、替代文本。
- 展示状态：至少区分草稿/隐藏与已发布；公开 `/photos` 只展示允许公开的照片。
- 排序：支持按排序值和发布时间/拍摄时间形成稳定顺序；同排序值时必须有确定的兜底排序。
- 上传/登记：首期可支持登记已有 URL 或模拟上传；未来 COS 接入时应通过服务端签名或服务端上传流程完成，前端不得直连长期密钥。
- 校验：必填字段、URL/Object Key 格式、图片类型、大小上限、状态值和排序值需要校验；错误应定位到字段或给出全局错误。
- 成功失败提示：创建、编辑、状态切换、排序、删除/隐藏、上传/登记完成后都需要明确反馈；失败时保留用户已输入内容，避免静默丢失。
- 权限：后台照片管理只对管理员开放；未登录或无权限访问必须有明确处理。

## COS 未来对接约束

- 服务端持有 COS `SecretId`、`SecretKey`、Bucket、Region 等敏感配置，前端不得保存或展示长期密钥。
- 前端上传若需要直传，必须通过服务端签发临时凭证或签名策略；签名应有明确有效期和允许操作范围。
- 公开展示图片时，页面只消费服务端返回的可访问 URL；若使用私有桶，URL 应支持过期时间并在过期后重新获取。
- 访问失败降级：签名过期、对象不存在、COS 网络失败或权限失败时，公开页面展示占位图/错误提示，后台展示可诊断错误，不泄露密钥或内部配置。
- 存储适配：需求要求业务层保留“媒体资源/照片记录”概念，不把 COS SDK 调用写死在 C 端页面中，便于未来从本地/静态 URL 切换到 COS。
- 缓存约束：后续设计需明确签名 URL 有效期、浏览器/CDN 缓存策略和刷新策略，避免 URL 过期后页面长期展示坏图。

## 文章详情 Markdown 需求

- 支持标准 Markdown 与 GFM，包括标题、段落、粗体/斜体、列表、任务列表、引用、表格、分隔线、链接和图片。
- 支持围栏代码块，识别语言标识并进行代码高亮；无语言标识、未知语言或高亮失败时降级为普通代码块。
- 代码块需要可读的排版、横向滚动或换行策略，不能撑破移动端布局；可选支持复制按钮，但首期不是强制项。
- 支持文章目录，根据标题层级生成锚点；桌面端可展示侧边目录，移动端可收敛为按钮/抽屉或文章顶部目录。
- 图片渲染需要支持响应式宽度、替代文本、加载失败占位和外链安全策略。
- 链接渲染需要区分站内/站外链接；站外链接应具备安全属性策略，避免不安全跳转行为。
- 安全 HTML 策略：默认不允许原始 HTML 直接执行；如未来确需支持 HTML，必须经过白名单和清洗策略后再开放。
- Markdown 渲染不应破坏现有文章元信息、标签入口、发布时间和阅读排版。
