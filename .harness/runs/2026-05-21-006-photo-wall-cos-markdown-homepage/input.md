# 006 照片墙与内容体验增强规划

## 背景

当前博客 MVP 已具备公开文章、标签、分类与后台内容管理基础，但下一轮体验需要更聚焦“IT 工程师个人技术博客”的信息表达：

- C 端首页应更加简洁清晰，首屏和主内容只突出个人介绍、文章与标签，减少杂乱区块和过多装饰。
- 需要新增照片墙能力，以瀑布流方式呈现项目现场、工作台、生活记录或技术活动图片。
- 照片资源后续计划对接腾讯云 COS，不应把存储实现写死在前端或业务页面里。
- 文章详情需要支持更完整的 Markdown 阅读体验，尤其是标题层级、表格、列表、引用、链接、图片、代码片段和代码高亮。

## 用户目标

用户希望先完成详细需求规划，不进入开发：

- 新增“照片墙”功能，C 端使用瀑布墙/瀑布流方式展示。
- 后续可对接腾讯云对象存储 COS，用于图片上传、访问与管理。
- 前端交互保持简洁、清晰、专业。
- 首页主要包含个人介绍、文章、标签。
- 文档详情需要支持 Markdown 与代码片段能力。
- 本阶段先分配 PM、Requirements、Architect agent 做需求细化、验收标准和设计方案。

## Skill 与文档依据

本轮规划应使用并体现以下本地 skill：

- `information-architecture`：梳理首页、文章、标签、照片墙、详情页与后台管理的信息架构和用户流。
- `frontend-design`：定义专业、克制、简洁的 C 端视觉与交互方向，避免区块堆叠和杂乱线条。
- `tailwind-design-system`：约束 C 端 Tailwind CSS v4 的 token、spacing、typography、responsive pattern。
- `frontend-architecture`：约束前端页面、组件、状态、数据加载与未来可扩展边界。
- `security-review`：上传、签名 URL、COS 凭证、管理员权限与文件校验相关风险需要进入规划。

本轮规划还需参考当前文档：

- Tencent Cloud COS Node.js SDK：规划后端存储适配、上传、对象访问 URL、签名 URL/临时访问策略。
- `react-markdown` / GFM：规划文章 Markdown 渲染、表格、任务列表、链接和安全 HTML 策略。
- Shiki：规划代码片段高亮、主题和服务端/构建期渲染策略。

## 目标

- 输出照片墙的产品需求、信息架构、页面/路由范围、后台管理范围和验收标准。
- 输出未来 Tencent Cloud COS 对接的技术边界：配置、凭证、上传流、访问 URL、缓存、回滚和安全约束。
- 输出首页简化方案：个人介绍、文章列表/精选文章、标签入口，减少无关模块。
- 输出文章详情 Markdown 能力规划：GFM、代码块、高亮、目录、图片、链接、安全策略。
- 输出后续开发拆分方案，明确需要 frontend、backend、database、tester、reviewer 等 agent 在开发阶段分别负责什么。

## 非目标

- 本阶段不修改 `apps/` 业务代码。
- 本阶段不新增真实 COS 密钥、环境变量或生产配置。
- 本阶段不执行数据库迁移、不上传真实图片、不改 CI/CD。
- 本阶段不写入 `docs/product/`，确认和 release 后再沉淀。
- 本阶段不启动开发、测试、评审、发布 agent。

## 初始需求拆分

### C 端公开页面

- 首页 `/`：聚焦个人介绍、文章、标签；可保留少量照片墙入口但不抢主视觉。
- 文章列表 `/articles`：保持清晰的文章浏览能力。
- 文章详情 `/articles/[slug]`：增强 Markdown、代码片段、目录和阅读排版。
- 标签页 `/tags/[slug]`：作为首页标签入口后的聚合页。
- 照片墙 `/photos`：瀑布流展示，支持分类/标签/年份筛选的规划。
- 照片详情：可规划为 `/photos/[id]` 或列表内轻量弹层，但需要明确移动端体验。

### 后台管理

- 照片管理：上传/登记图片、标题、描述、标签、拍摄时间、展示状态、排序和封面控制。
- 存储管理：先支持本地/模拟 URL 或已有静态资源，后续通过存储适配器切换 COS。
- Markdown 管理：文章编辑时明确 Markdown 输入、预览、代码块语言标识和图片引用方式。

### API 与数据

- 新增照片/媒体资源的数据模型规划。
- 新增公开照片列表、照片详情、后台照片 CRUD、上传签名/上传完成回调等 API 规划。
- 明确 COS 对接不让前端持有 SecretId/SecretKey，敏感凭证只留在服务端或使用临时凭证。

## 验收标准

- [ ] `artifacts/requirement.md` 细化用户角色、使用场景、页面范围、后台范围、功能优先级、非目标和验收标准。
- [ ] `artifacts/architecture.md` 说明 C 端 IA、视觉方向、组件边界、API 边界、存储抽象、COS 安全策略和风险。
- [ ] `artifacts/implementation-plan.md` 明确开发阶段任务拆分、涉及文件/模块、顺序、人工确认点、测试计划和完成检查。
- [ ] `artifacts/api-change.md` 说明是否涉及 API，并列出规划中的公开和后台接口契约草案。
- [ ] `artifacts/db-migration.md` 说明是否涉及数据库，并列出规划中的表/字段/索引/迁移风险草案。
- [ ] 方案明确说明首页只保留个人介绍、文章、标签作为核心内容层级。
- [ ] 方案明确说明照片墙瀑布流的加载、空态、错误态、图片比例、移动端布局和性能要求。
- [ ] 方案明确说明 Markdown 支持范围，包括 GFM、代码块语言、代码高亮、安全 HTML 策略。
- [ ] 本阶段不修改业务代码、不写 `docs/product/`、不进入开发。

## 测试要求

- 规划阶段运行 harness 结构检查。
- 开发阶段需要覆盖：
  - C 端首页桌面/移动端视觉截图验收。
  - 照片墙瀑布流桌面/移动端布局、加载、空态、错误态、点击详情。
  - Markdown 文章详情对标题、表格、列表、引用、链接、图片、代码块的渲染。
  - 后台照片管理的鉴权、表单校验、成功/失败提示、发布状态。
  - API 与数据库迁移的回归测试。

## 回滚方式

- 规划阶段仅新增 `.harness/backlog/ready/` 和 `.harness/runs/<run>/artifacts/` 内容，不涉及业务回滚。
- 若方案不满意，可在当前 run 中重派 PM / Requirements / Architect agent 修订。
- 开发阶段需以 feature scope 隔离照片墙、Markdown 渲染和 COS 存储适配，保留回退到现有文章和首页体验的路径。
