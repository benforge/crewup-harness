# C 端个性简约风格探索需求扩写

> 已由用户确认继续推进。本文已从 `requirement-plan.md` 提升为正式 `requirement.md`，后续进入架构与实施计划阶段；实现仍需在方案产物汇总后再次确认。

## 原始需求摘要

用户希望继续打磨 C 端公开站点的视觉风格：要有个性，但整体保持简约；当前还没有完全想清楚方向，需要先把审美方向、边界和验收标准收敛出来。

## 背景

现有 C 端已经完成过几轮专业化调整，定位从普通博客收敛为“IT 工程师技术站 / 工程师手记”。当前页面整体更克制、内容优先，已经减少了卡片、线条和营销感，但风格辨识度仍可继续加强。

本轮不是“再好看一点”的泛化调样式，而是要回答一个更具体的问题：这个站点如何在保持阅读舒适、专业可信的前提下，让读者一眼记住它是一个有作者气质的技术站。

需求扩写阶段不改业务代码；用户确认后进入架构与实施计划阶段。正式写业务代码前仍需确认 implementation gate。

## 目标

- 明确 C 端“个性但简约”的主风格方向，避免后续实现阶段在颜色、字体、布局、照片和动效上各自发挥。
- 在现有“IT 工程师技术站”基础上增加可记忆的视觉语言，而不是推翻已有专业阅读体验。
- 定义可截图验收的视觉标准，覆盖首页、文章列表、文章详情、关于页、照片入口、404/空态。
- 明确哪些个性元素可以使用，哪些会破坏简约感和专业可信度。
- 为后续 architect / frontend agent 提供足够清晰的方案输入。

## 非目标

- 本阶段不修改 `apps/` 业务代码。
- 本阶段不修改 Admin、API、数据库、infra、CI/CD 或生产配置。
- 不做营销落地页、作品集式炫技、重动效页面或视觉噱头堆叠。
- 不把个性化简化为换一套颜色、换字体或增加大量装饰。
- 不牺牲文章详情页的连续阅读体验。
- 不把照片墙提升为首页主叙事，除非用户后续明确选择这个方向。

## 用户角色与权限

- 公开读者：未登录访问 C 端，主要阅读文章、浏览归档、查看关于页和照片内容。关注站点是否可信、内容是否清晰、阅读是否舒服。
- 回访读者：通过分类、标签、搜索或照片入口持续探索内容。关注视觉一致性、导航稳定性和内容识别效率。
- 博主 / 站点维护者：希望站点体现个人工程气质，但不显得花哨、幼稚或像模板站。
- 权限边界：本需求只涉及公开 C 端展示，不新增登录态、权限、管理端流程或受保护路由。

## 核心用户流程

- 首页进入：读者在首屏判断站点定位和作者气质，看到一个明确的主要阅读入口。
- 文章浏览：读者在列表页快速扫描标题、摘要、时间、分类和标签，不被装饰元素打断。
- 文章阅读：读者进入详情页后专注阅读正文，代码、引用、注释和图片具备统一的工程笔记感。
- 关于判断：读者通过关于页理解作者的写作边界、技术方向和站点性格。
- 照片探索：读者通过照片墙获得轻量真实感，照片用于补充工程现场和工作痕迹，不抢文章主线。
- 异常恢复：读者遇到 404、空态或 fallback 内容时，仍能感到站点语言和视觉一致。

## 页面与路由候选

- 首页 `/`：本轮最重要页面。需要承载主风格第一印象、作者气质、重点文章入口和照片/关于的轻量线索。
- 文章列表 `/articles`：保持扫描效率，增加一点技术刊物/工程索引感。
- 文章详情 `/articles/[slug]`：优先保证阅读，个性主要体现在标题区、正文排版、代码块、引用、阅读 note 和上下篇导航。
- 分类页 `/categories/[slug]` 与标签页 `/tags/[slug]`：复用列表语言，不新增强装饰。
- 关于页 `/about`：适合强化“作者气质”和写作边界，可比文章列表更有个性。
- 照片墙 `/photos` 与照片详情 `/photos/[id]`：保留次级入口，可作为真实工作痕迹的辅助风格来源。
- 404 / 空态 / 错误态：使用同一套语气和简约视觉规则，避免默认模板感。

## 数据与接口线索

- 本轮核心是视觉与信息表达，原则上不需要 API schema、数据库结构或后端接口变化。
- 现有文章、分类、标签、照片数据结构应尽量复用。
- 如果后续需要更强的个性内容，优先通过已有字段表达：文章摘要、分类说明、标签、照片标题/描述、关于页文案。
- 不在本轮要求新增主题配置、站点设置接口、媒体元数据字段或用户偏好字段。

## 候选风格方向

### 方向 A：工程手札

- 核心气质：一个工程师长期维护的工作手记，干净、真实、略带手工痕迹。
- 视觉关键词：纸面、注释、标注线、代码旁注、阅读笔记、低饱和强调色。
- 个性来源：标题和元信息像手记条目，正文里有克制的 note / quote / code 细节，照片像工作现场补充。
- 简约边界：装饰只用于解释结构，不出现大面积插画、复杂背景或多层卡片。
- 适合程度：高。最贴合“工程师手记”和当前站点基底。
- 主要风险：标注、纸感和线条如果过多，会回到“区块线条杂乱”的旧问题。

### 方向 B：冷静编辑部

- 核心气质：独立技术刊物 / 小型工程编辑部，克制、精准、有编辑判断。
- 视觉关键词：严谨网格、强标题、章节编号、精选文章、稳定节奏。
- 个性来源：首页像技术刊物目录，文章标题和摘要更有编辑层级，列表像清晰的 issue / archive。
- 简约边界：不做杂志式大图轰炸，不堆多个封面模块。
- 适合程度：中高。专业感强，但个人温度需要通过关于页、文案和照片补足。
- 主要风险：容易太冷、太像机构媒体，削弱个人站气质。

### 方向 C：轻量实验室

- 核心气质：技术实验台 / 原型记录 / 工具方法库，简约但有探索感。
- 视觉关键词：轻网格、模块编号、状态标签、实验记录、局部高亮、照片现场感。
- 个性来源：首页和归档页像工程实验记录，照片墙和状态标签更有存在感。
- 简约边界：不把 C 端做成 dashboard，不使用过密指标、不引入后台感。
- 适合程度：中。更有个性，但更容易偏工具产品。
- 主要风险：信息密度过高，影响普通读者阅读。

## 推荐方向

建议主方向选择“工程手札”，吸收“冷静编辑部”的排版秩序。

推荐表达：

- 不是普通模板博客，而是一份持续维护的工程手记。
- 不是企业官网，也不是作品集，而是有清晰作者判断的技术记录。
- 个性来自真实、克制、可读的细节：标题排版、注释感、代码块、照片入口、关于页语气。

推荐视觉策略：

- 首页首屏保留一个主焦点，避免多个强区块竞争。
- 背景可保留轻网格，但需要更像纸面/工程标注，而不是纯装饰纹理。
- 色彩继续低饱和，但可增加一个更稳定的“作者识别色”，用于链接、注释、焦点线和选中态。
- 字体保持中文可读优先，可考虑标题和正文形成更明确的性格差异。
- 线条只用于必要结构，优先用留白和字体层级区分内容。
- 照片墙保留次级入口，但首页可出现一条非常轻的“现场感”线索。

## 验收标准

- [ ] AC-1：首页首屏能在 5 秒内传达“有作者气质的 IT 工程师技术站”，且只有一个最高视觉焦点。
- [ ] AC-2：页面整体保持简约，首页、列表、详情页不得出现装饰堆叠、卡片套卡片、过量边框或多个互相竞争的强背景。
- [ ] AC-3：文章详情页阅读体验不下降，正文宽度、段落间距、代码块、引用和图片在桌面与移动端均稳定可读。
- [ ] AC-4：个性元素在至少首页、关于页、文章详情页中形成一致语言，而不是单页孤立效果。
- [ ] AC-5：照片入口可以提供真实工作痕迹，但不得抢占文章作为主线内容的优先级。
- [ ] AC-6：移动端无横向滚动、文字遮挡、按钮溢出、标签挤压和导航遮挡。
- [ ] AC-7：404、空态、fallback 提示延续同一套语气与视觉细节，不出现默认模板感。
- [ ] AC-8：后续实现前必须提供桌面与移动截图验收清单，至少覆盖首页、文章列表、文章详情、关于页和 404/空态。

## 风险与边界

- 风格过轻：如果只换颜色或微调间距，读者感知不到“个性”。
- 风格过重：如果加入太多标注、纹理、动效或照片，可能破坏当前已经改善的简约阅读体验。
- 中英文混杂：当前 C 端已有英文导航和部分英文标题，后续需要确认是保留技术感，还是更中文化。
- 照片权重：照片墙能增强真实感，但如果进入首页主叙事，可能改变“文章为主”的产品定位。
- 实现边界：本阶段不进入开发；用户确认后，需要先由 architect 产出视觉系统方案和实施计划，再启动 frontend 实现。
- 委派降级：项目 harness 期望 requirements-plan 由子 agent 执行；当前会话未获得用户明确“使用子 agent/委派”的授权，因此本次由主 agent 代为扩写需求，并在最终汇总中说明。
