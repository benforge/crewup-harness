# 需求说明

## PM 入口判断

- 需求性质：本轮属于正式 UI 框架化与体验优化迭代，需要拆分为 requirements、architect、frontend、tester、reviewer 阶段；PM 仅确认范围、优先级、阶段边界与待确认问题，不进入实现。
- 优先级：P1。优先解决 Admin 操作反馈不完整、基础组件不一致、C 端样式维护分散与整体视觉层级混乱问题；不扩大为全站重设计。
- 本轮范围：Admin 侧确认引入 Ant Design 作为企业后台 UI 基础，覆盖布局、导航、表单、表格、弹窗/确认、消息提示、加载、空状态与错误状态；Web C 端确认引入 Tailwind CSS 作为样式组织与响应式基础，优先覆盖核心阅读路径、列表/详情、导航、基础页面布局与移动端适配；整体补齐成功、失败、加载、确认、空状态等交互反馈。
- 阶段边界：规划阶段只产出 requirement、architecture、implementation-plan 等 run artifacts；开发阶段再修改 Admin/Web 代码与依赖；测试阶段必须覆盖 Admin 路由/鉴权流、表单/表格操作反馈、C 端桌面与移动端视觉回归、构建验证。
- 非目标：不修改数据库 schema；不改 API 合约，除非后续架构确认 UI 状态确实缺少后端字段；不改生产部署、密钥和 CI/CD；不写入 docs/product；不做无边界的全站重构或品牌视觉重制。
- 建议拆分：第一阶段完成需求澄清和技术方案；第二阶段先接入框架与主题基础；第三阶段迁移 Admin 核心页面和反馈状态；第四阶段迁移 C 端核心阅读路径与响应式；第五阶段测试、评审和发布总结。
- 待确认问题：Admin 是否明确使用 Ant Design v5；Web C 端是否明确使用 Tailwind CSS v4；本轮必须调整的 Admin 页面清单；本轮必须调整的 C 端页面清单；是否保留现有视觉风格为主，只做统一和清爽化；是否需要深色模式、多语言或无障碍增强进入本轮；是否允许新增/升级前端依赖并更新 lockfile。

## 背景

当前 Admin 与 Web C 端已有基础闭环，但 UI 组件、反馈状态和响应式表现不够统一。本轮需求聚焦框架化接入前的可验收范围澄清，确保后续开发只围绕核心页面和关键交互做产品化优化。

## 目标

- Admin 可引入 Ant Design 作为后台 UI 基础，统一布局、导航、表单、表格、弹窗和反馈组件。
- Admin 补齐成功、失败、加载、确认、空态提示，减少无反馈或反馈不一致的操作。
- Web C 端可引入 Tailwind CSS 作为样式组织与响应式基础，让页面层级更清楚、布局更稳定、交互更友好。
- 明确本轮页面范围、验收标准和待确认问题，供 architect/frontend/tester/reviewer 后续使用。

## 非目标

- 不做全站品牌重设计，不重写业务流程。
- 不修改数据库 schema、生产配置、密钥、CI/CD。
- 不改 API 合约，除非后续架构确认 UI 状态缺少必要字段。
- 不在需求阶段修改 `apps/` 业务代码或新增依赖。
- 不写入 `docs/product/`。

## 用户角色与权限

- 未登录访客：只能访问 Web C 端公开页面；访问 Admin 时应被引导到登录或未授权处理。
- 普通读者：浏览 C 端列表、详情和基础页面；不进入 Admin。
- Admin 管理员：登录后访问 Admin 受保护路由，进行内容管理、表单提交、列表操作和退出登录。
- 未授权用户：访问 Admin 受保护页面时必须看到明确拦截或跳转，不暴露管理内容。

## 核心用户流程

- Admin 登录后进入后台首页，能通过清晰导航到达列表、编辑/新建、详情或设置类页面。
- Admin 执行保存、删除、发布、撤销等操作时，先看到必要确认，再看到加载、成功或失败反馈。
- Admin 列表无数据、筛选无结果或接口失败时，页面给出空态/错误态和可执行下一步。
- C 端访客从首页/列表进入文章详情，能清晰识别主内容、导航、返回路径和移动端布局。
- C 端在加载、空列表、文章不存在或请求失败时，界面不塌陷，并提供明确提示。

## 页面与路由清单

- Admin 必调：登录页、后台布局/导航、内容列表页、内容新建/编辑页、详情/预览相关页。
- Admin 兼容检查：受保护路由、未授权/登录失效处理、退出登录入口。
- Web 必调：首页/文章列表、文章详情、基础导航、空列表/404 或不存在页面。
- Web 兼容检查：桌面、平板、移动端主要断点；长标题、长正文、空数据和加载中页面。

## Admin 反馈状态清单

- 成功：保存、新建、更新、删除、发布等操作完成后显示统一成功提示。
- 失败：接口错误、校验失败、权限不足、网络异常时显示错误提示，并保留用户已输入内容。
- 加载：页面初始加载、表格刷新、表单提交、按钮级异步操作有明确 loading 状态。
- 确认：删除、覆盖、退出未保存编辑等高影响操作必须二次确认。
- 空态：列表为空、筛选无结果、详情不存在时使用统一空态，并提供返回、重置筛选或新建入口。

## Web 视觉与响应式验收

- 信息层级：主标题、正文、元信息、操作入口有明显层级，不依赖大段自定义 CSS 堆叠。
- 布局清晰：页面留白、间距、对齐统一；列表与详情的阅读宽度稳定。
- 交互友好：链接、按钮、返回、加载和错误状态可识别；移动端可点击区域不拥挤。
- 响应式：桌面、平板、手机下无横向滚动、内容遮挡、按钮文字溢出或导航重叠。
- 不可接受：颜色/字号混乱、卡片层层嵌套、移动端正文过宽、空白页面无提示。
