# 需求扩写草案
> 这是 plan-only 阶段产物。用户确认后，才可以提升为正式 `requirement.md` 并进入后续流程。

## 原始需求摘要
- 把 `web` 项目里已经开发好的页面全部重构 CSS 写法，统一改成 Tailwind CSS。
- 现有页面的样式实现方式收敛到 Tailwind utility class，不再沿用分散的自定义 CSS 写法。
- 后续新增页面默认优先用 Tailwind-first。
- 不改业务逻辑，只做样式写法重构和相关规则调整。

## 需求背景
- 当前 `apps/web` 已经使用 Next.js App Router、React 和 Tailwind CSS v4，页面里同时存在局部自定义样式与 utility class。
- 该项目的既有规则已经明确：新页面/新组件优先使用 Tailwind utility class，`app/globals.css` 只保留入口、`@theme` token、base 和少量跨页样式，不应继续堆页级布局、卡片、列表或按钮样式。
- 本次需求的目标不是再讨论技术选型，而是把 `web` 现有页面统一收敛到 Tailwind-first 的实现方式，让页面样式写法保持一致、可维护、可扩展。
- 当前页面包含首页、文章、照片、分类、标签、关于页，以及错误/加载/空态等状态页，迁移时需要覆盖这些既有页面的主要视觉实现。

## 过往背景
- 相关历史 run：`2026-05-19-003-ui-framework-polish` 已沉淀 admin 侧 Ant Design v5 主题与受保护路由基线，但本次不纳入 admin。
- 复用的历史背景：web 侧已经确立 Tailwind utility-first 方向，本次只是把既有页面样式实现方式统一收敛。
- 冲突或变化：本次从“web + admin 逐步迁移”收窄为“仅 web 全量重构”，且明确不改业务逻辑。

## 目标
- 让 `web` 现有页面的样式实现方式统一收敛到 Tailwind CSS。
- 让 `web` 后续新增页面默认采用 Tailwind-first 写法，不再延续旧式分散 CSS 方案。
- 让页面布局、间距、宽度、对齐、响应式、空态、错误态和主要排版都能用 Tailwind utility 直接表达。
- 在不改变业务逻辑和数据流的前提下，完成已有页面的样式写法重构。

## 非目标
- 不改业务逻辑、接口契约、数据模型、路由结构或内容生成规则。
- 不做新的架构选型，不引入新的 CSS 框架，也不把这次需求扩展到非 `web` 项目。
- 不把 `admin` 纳入本次范围。
- 不把 `app/globals.css` 继续当作页级样式堆积区。
- 不写 `docs/product/`，也不在本阶段输出长期产品沉淀文档。

## 用户角色与权限
- 普通访客：访问 `web` 的公开内容页面，浏览文章、分类、标签、照片和关于页。
- 未登录用户同样可以访问公开页面，权限边界不因样式重构而变化。
- 本次需求不涉及后台用户、管理端权限或受保护工作区。

## 核心用户流程
- 访客打开首页，查看首屏、近期文章、标签入口和照片入口。
- 访客进入文章列表，搜索、翻页，再进入文章详情。
- 访客进入照片页，按标签、分类、年份筛选照片并查看空态或错误态。
- 访客进入分类页或标签页，继续浏览相关归档内容。
- 访客进入关于页，查看站点与作者说明。
- 所有流程在桌面端和移动端都要保持可读、可操作、无明显布局断裂。

## 页面与路由候选
- `/`：首页首屏、文章引导、标签入口、照片入口。
- `/articles`：文章归档、搜索、分页、分类/标签入口。
- `/articles/[slug]`：文章详情、正文排版、阅读信息、相关跳转。
- `/photos`：照片墙、筛选面板、网格/瀑布流、空态。
- `/photos/[id]`：照片详情与上下文展示。
- `/about`：作者与站点说明。
- `/categories/[slug]`：分类归档页。
- `/tags/[slug]`：标签归档页。
- `not-found`：找不到页面时的兜底视图。
- `loading`、`error`：页面级加载与错误状态。
- `sitemap`、`robots`：基础站点索引与抓取相关路由，样式上保持现有一致性。

## 数据与接口线索
- 环境变量线索：`BLOG_API_BASE_URL`、`NEXT_PUBLIC_API_BASE_URL`，默认指向 `http://localhost:3001`。
- 文章相关接口：`GET /api/articles`、`GET /api/articles/:slug`、`POST /api/articles/:slug/view`。
- 分类/标签接口：`GET /api/categories`、`GET /api/tags`。
- 照片接口：`GET /api/photos`、`GET /api/photos/:id`。
- 现有 `apps/web/lib/api.ts` 已经包含 fallback 数据、加载状态和归一化逻辑，样式重构应优先复用这些现成数据线索来验证视觉结果，不改变数据结构本身。

## 风险与边界
- `Tailwind CSS v4` 依赖完整 class token 扫描，重构时不能用动态拼接 class 代替真实 class 映射。
- `web` 的 `globals.css` 已经承担了入口和少量跨页样式，继续塞入页级样式会让 Tailwind-first 规则失焦。
- 全量迁移容易引入响应式回归，尤其是小屏下的换行、宽度、间距和对齐问题，需要重点检查。
- 页面里存在 fallback、空态和错误态，样式重构不能只看正常数据态。
- 如果局部样式仍然保留，必须有清晰边界，避免把旧 CSS 继续扩散成新的默认做法。
