# 需求说明：IT 工程师博主体验专业化重设计

## 背景

当前 C 端博客与 Admin 后台已经具备基础页面和管理方向，但整体体验仍缺少“IT 工程师博主”的专业辨识度。C 端容易落入泛模板博客或营销落地页的表达，Admin 容易变成表单和功能入口的堆叠。本轮只做规划，不开发，目的是先明确产品定位、两条体验线的需求范围、优先级、边界和人工审核点，为后续 Requirements / Architect / Frontend 等 agent 提供可执行的判断依据。

## 判断依据

- `frontend-design`：用于判断 C 端和 Admin 是否有清晰、克制、可记忆的视觉方向，避免 generic AI aesthetics、过度营销化 hero、同质化卡片和模板感。
- `web-design-guidelines`：用于后续人工审核可用性、可访问性、布局稳定性、文字可读性、交互反馈和错误状态，不把“好看”当作唯一验收标准。
- `information-architecture`：用于先定义站点/后台的导航、页面层级、内容优先级、用户流和 URL/路由策略，再进入视觉设计或开发。

## 产品定位

本项目定位为“IT 工程师博主”的专业技术博客与内容管理工作台。

C 端应像一个可信的个人技术站点，服务读者快速理解博主的技术方向、找到高质量文章、判断内容可信度，并持续阅读工程经验、技术文章、项目复盘、工具方法和架构思考。它不是营销页，不以转化漏斗、夸张口号、装饰性 hero 或品牌广告感为核心；也不是泛模板博客，不以默认文章卡片、杂乱分区和无差别标签堆叠作为主要体验。

Admin 应像一个高效、可靠的 CMS 工作台，服务博主持续生产、组织、发布和维护内容。它不是把所有功能放进单页表单的集合，也不是仅为展示 UI 框架而存在的后台模板。

## 目标

- 明确 C 端公开技术博客的页面范围、信息层级、内容气质和验收标准。
- 明确 Admin CMS 工作台的路由层级、鉴权态、导航、表格、表单、反馈和异常状态需求。
- 把“IT 工程师博主”的定位转成后续开发 agent 可执行的视觉、文案、交互和验收要求。

## 用户与价值

- C 端读者：希望快速判断博客是否值得读，能按主题、时间、标签或搜索找到有用内容，并在文章详情页获得稳定、专注、可阅读的技术阅读体验。
- 博主/管理员：希望高效管理文章、分类、标签、媒体和基础设置，清楚看到内容状态，获得可靠的操作反馈，减少发布和维护过程中的不确定性。
- 后续开发 agent：需要明确哪些是 P0 必须完成的体验骨架，哪些是增强项，哪些明确不做，避免把重设计误解成单纯换皮。

## C 端体验线需求拆分

### P0

- 首页必须建立专业技术博客的第一印象：清晰说明博主身份、技术方向、重点内容入口和近期/精选文章，不使用营销落地页式夸张卖点。
- 文章列表必须支持清晰的内容浏览：标题、摘要、发布时间、分类/标签、阅读线索和分页/加载策略要有明确层级。
- 文章详情必须优先保障阅读体验：标题区、元信息、正文排版、代码块、目录/锚点、上一篇/下一篇或相关文章入口需要有明确策略。
- 分类与标签必须服务内容发现：区分“分类”作为主题归档，“标签”作为横向索引，避免无意义标签云。
- 搜索、空状态、404 必须具备可信的基础体验：告诉用户发生了什么、还能去哪里、如何继续找内容。
- 移动端与桌面端都必须保持可读、可导航、无明显遮挡或溢出。

### P1

- 关于页补充博主专业背景、技术栈、项目经历和写作主题边界，建立可信度但不做简历营销页。
- 首页可增加“系列文章”“工程实践”“项目复盘”等面向 IT 工程师的内容分组。
- 文章详情可增强目录定位、代码块复制、阅读进度、相关推荐等技术阅读辅助能力。
- 列表页可增加按分类、标签、时间的筛选组合，但需要保持克制，不把筛选 UI 做成复杂后台。

### P2

- 订阅、RSS、主题切换、文章收藏、评论、阅读统计等增强能力可作为后续迭代，不进入本轮核心范围。
- 专题页、项目页、工具箱页可在内容规模足够后再规划。
- 个性化推荐、AI 摘要、多语言等能力暂不作为本轮体验重设计的必要条件。

## Admin 体验线需求拆分

### P0

- 登录与鉴权态必须清晰：未登录、登录中、无权限、会话失效和登录失败要有明确反馈与去向。
- 全局布局必须像完整工作台：侧边导航、顶部区域、面包屑/当前位置、主内容区和基础响应式策略要清楚。
- 内容管理必须覆盖文章列表、创建/编辑、状态管理、保存、发布、删除/撤销确认等核心流程。
- 分类/标签管理必须与 C 端发现逻辑一致：支持新增、编辑、删除前确认、被文章引用时的边界提示。
- 表格、表单、弹窗、通知、空状态、错误状态必须形成一致模式，不能每页各自发挥。
- 操作反馈必须可靠：保存成功、保存失败、校验错误、网络错误、权限错误要可辨认。

### P1

- 媒体管理支持基础上传/选择/替换/删除确认，服务文章封面与正文图片，不做复杂素材库。
- 设置页支持站点名称、博主信息、社交链接、SEO 基础字段等内容配置。
- Dashboard 可展示内容概况、待处理草稿、最近更新和快速入口，但不做数据分析产品。
- 编辑流程可增加预览、草稿自动保存提示、离开未保存确认等生产效率能力。

### P2

- 多用户协作、角色权限矩阵、审批流、复杂版本管理、内容定时发布、审计日志、批量导入导出等暂列后续增强。
- 高级统计、增长分析、A/B 测试、营销活动管理不属于本轮 Admin 核心。

## C 端页面级需求

### 全局页面原则

- 核心任务：让读者快速判断“这里是否有可信的工程内容”，并能在 1-2 次导航内进入文章、分类、标签或关于页。
- 信息优先级：技术身份与内容方向优先于装饰；文章标题、主题、时间、摘要优先于视觉噱头；正文阅读优先于推荐和辅助模块。
- 主要交互：顶部导航、文章入口、分类/标签入口、分页或加载更多、文章内锚点、上一篇/下一篇或返回列表。
- 状态要求：加载态要稳定占位；空态要解释原因并给出回到文章列表/首页/分类的路径；错误态要避免技术堆栈暴露。
- 验收标准：桌面与移动视口均无文字溢出、导航遮挡、按钮误触区过小、卡片高度跳动或正文阅读宽度失控。

### 首页 `/`（P0）

- 核心任务：建立 IT 工程师博主的可信第一印象，引导读者进入重点文章、最新文章、分类和关于页。
- 信息优先级：1. 博主身份与技术方向；2. 精选/近期文章；3. 主要内容主题，如工程实践、架构思考、项目复盘、工具方法；4. 关于入口和归档入口。
- 主要交互：首页导航跳转、精选文章点击、主题分组点击、查看全部文章、移动端导航展开/收起。
- 成功态：读者能在首屏理解“作者是谁、写什么、为什么可信”，并看到至少一个明确文章入口。
- 失败态：文章数据加载失败时展示克制错误提示，提供重试和进入文章列表入口。
- 空态：暂无文章时首页不展示空洞卡片堆叠，应展示写作方向、即将发布的主题和关于页入口。
- 验收标准：首页不得呈现营销落地页式大口号、夸张转化按钮或无内容支撑的装饰 hero；首屏需要露出下一段内容线索。

### 文章列表 `/articles`（P0）

- 核心任务：帮助读者按时间和主题浏览全部文章，并快速判断每篇文章是否值得打开。
- 信息优先级：1. 页面标题和文章总览说明；2. 文章标题；3. 摘要/导语；4. 发布时间、分类、标签、阅读时间等元信息；5. 分页/加载更多。
- 主要交互：点击文章进入详情，按分类/标签跳转，分页或加载更多，移动端保持单列可读。
- 成功态：文章列表可扫描，元信息层级一致，读者不需要进入详情也能判断文章主题。
- 失败态：列表请求失败时提示无法加载文章，并提供重试。
- 空态：暂无文章时说明当前还没有公开文章，提供返回首页或查看关于页。
- 验收标准：不得把列表做成同质化营销卡片墙；列表密度应服务阅读扫描，分页/加载策略必须明确。

### 文章详情 `/articles/[slug]`（P0）

- 核心任务：提供专注、稳定、适合技术内容的长文阅读体验。
- 信息优先级：1. 标题；2. 发布时间、分类、标签、阅读时间；3. 摘要或导语；4. 正文；5. 目录/锚点；6. 上一篇/下一篇或相关文章。
- 主要交互：目录锚点跳转、分类/标签跳转、代码块横向滚动或复制（复制可 P1）、上一篇/下一篇跳转、返回列表。
- 成功态：正文排版清晰，代码块、引用、列表、标题层级和图片都稳定可读。
- 失败态：文章不存在或未发布时进入 404 或文章不可用状态，不显示半截正文。
- 空态：文章正文为空时显示“内容尚未完善”的编辑/发布异常提示，避免空白详情页。
- 验收标准：正文最大阅读宽度受控；代码块不撑破布局；目录不会遮挡正文；移动端标题、代码、表格不横向破坏整页。

### 分类归档 `/categories/[slug]`（P0）

- 核心任务：按主题归档文章，帮助读者理解该分类下的内容边界。
- 信息优先级：1. 分类名称；2. 分类说明；3. 文章数量或文章列表；4. 相关标签或返回全部分类/文章入口。
- 主要交互：打开文章、跳转其他分类、返回文章列表。
- 成功态：分类页能说明该主题聚焦什么，并展示属于该主题的文章。
- 失败态：分类不存在时进入 404 或“分类不存在”状态，提供返回文章列表。
- 空态：分类存在但暂无文章时说明暂无公开文章，提供浏览全部文章。
- 验收标准：分类必须表现为主主题归档，不做无意义分类标签混排。

### 标签归档 `/tags/[slug]`（P0）

- 核心任务：按横向技术关键词索引文章，支持读者围绕某个工具、框架、方法或概念继续阅读。
- 信息优先级：1. 标签名称；2. 标签使用说明或文章数量；3. 相关文章列表；4. 相关分类入口。
- 主要交互：打开文章、跳转相关分类或全部文章。
- 成功态：标签页聚合具有共同关键词的文章，且列表呈现方式与文章列表一致。
- 失败态：标签不存在时进入 404 或“标签不存在”状态，提供返回文章列表。
- 空态：标签存在但暂无公开文章时说明暂无内容，并给出全部文章入口。
- 验收标准：标签不得以大面积标签云作为主要体验；标签命名要服务工程检索，如框架、语言、工具、实践方法。

### 关于页 `/about`（P0，内容可轻量）

- 核心任务：解释博主的专业背景、技术关注点、写作边界和联系/社交入口，增强可信度。
- 信息优先级：1. 身份定位；2. 技术栈/工程方向；3. 写作主题；4. 项目或经验线索；5. 联系方式或社交链接。
- 主要交互：跳转文章列表、分类、外部社交链接。
- 成功态：读者能判断作者为什么有资格写这些技术内容。
- 失败态：外链不可用时不影响页面主体阅读。
- 空态：若暂无完整个人资料，至少展示技术方向、写作主题和文章入口。
- 验收标准：关于页不做简历营销页，不堆砌浮夸形容词，不把个人展示凌驾于内容可信度之上。

### 404（P0）

- 核心任务：在路由不存在时让读者知道发生了什么，并继续发现内容。
- 信息优先级：1. 页面不存在说明；2. 返回首页；3. 查看文章列表；4. 可选分类/标签入口。
- 主要交互：返回首页、进入文章列表、浏览分类或标签。
- 成功态：错误页语气克制，和站点整体一致。
- 失败态：不得暴露框架错误、堆栈、调试信息。
- 空态：不适用。
- 验收标准：直接访问不存在的文章、分类、标签或任意路径时，都有明确去向。

### 空态（P0）

- 核心任务：在首页、列表、分类、标签、搜索或详情缺少内容时保持专业感和可继续行动。
- 信息优先级：1. 当前没有什么；2. 为什么可能为空；3. 下一步去哪；4. 可选重试。
- 主要交互：返回首页、查看全部文章、浏览分类、重试加载。
- 成功态：空态不让页面像坏掉了，也不使用花哨插画替代信息。
- 失败态：网络错误、权限/发布状态错误、数据为空要有不同文案，不混用“暂无数据”。
- 验收标准：所有 P0 页面必须有人工可见的空态策略。

## Admin 页面与流程级需求

### Admin 全局工作台原则

- 核心任务：让博主稳定完成内容生产、组织、发布和维护，降低误操作和状态不确定性。
- 信息优先级：当前任务和内容状态优先；导航、面包屑、操作反馈辅助定位；统计和装饰信息让位于编辑效率。
- 主要交互：侧边导航、面包屑、表格筛选/分页、表单校验、保存/发布/删除确认、消息通知。
- 状态要求：所有页面必须考虑加载、成功、失败、空态、无权限、会话失效；危险操作必须确认。
- 验收标准：Admin 应像 CMS 工作台，不像 Ant Design 组件演示页；同类表格、表单、弹窗、通知和空态必须一致。

### 登录 `/login`（P0）

- 核心任务：让管理员安全进入后台，并清楚处理登录失败或会话问题。
- 信息优先级：1. 后台名称/站点归属；2. 登录表单；3. 登录错误；4. 必要的返回站点入口。
- 主要交互：输入账号密码、提交登录、展示提交中状态、登录成功跳转 Dashboard、失败后可重试。
- 成功态：登录后进入 `/dashboard` 或原目标页。
- 失败态：账号密码错误、网络错误、会话失效重登都要有可辨认反馈。
- 空态：不适用，但表单初始态不得出现无意义占位内容。
- 验收标准：提交中按钮不可重复触发；错误提示不泄露敏感信息；未登录访问后台路由应跳转登录。

### Dashboard `/dashboard`（P0 骨架，P1 增强）

- 核心任务：提供后台进入后的方向感和常用入口。
- 信息优先级：1. 内容概况；2. 最近文章/草稿；3. 快速创建文章；4. 分类/标签管理入口。
- 主要交互：快速新建文章、进入文章列表、进入最近编辑项、进入分类/标签。
- 成功态：管理员能在 5 秒内找到下一步常用操作。
- 失败态：统计或最近内容加载失败时不影响导航和快速入口。
- 空态：没有文章时突出“新建文章”入口，并提示先建立分类/标签可选。
- 验收标准：本轮 P0 至少完成工作台骨架和核心入口；复杂趋势图、增长分析、流量统计为 P2。

### 文章列表 `/articles`（P0）

- 核心任务：管理全部文章，支持查找、识别状态、进入编辑和执行关键操作。
- 信息优先级：1. 新建文章入口；2. 文章表格；3. 状态、分类、更新时间、发布时间；4. 筛选/搜索；5. 批量能力（后续）。
- 主要交互：新建、编辑、按状态/分类筛选、搜索标题、分页、删除确认、发布/下线入口（按现有能力落地）。
- 成功态：文章状态一眼可辨，点击标题或编辑按钮进入 `/articles/:id`。
- 失败态：列表加载失败可重试；删除/发布失败要回显错误并保持原状态。
- 空态：暂无文章时展示新建文章主入口。
- 验收标准：表格列不能拥挤到移动端不可用；危险操作必须二次确认；状态标签命名与 C 端发布可见性一致。

### 新建文章 `/articles/new`（P0）

- 核心任务：创建一篇可保存为草稿或发布的文章。
- 信息优先级：1. 标题；2. slug；3. 摘要；4. 正文；5. 分类/标签；6. 发布状态和保存动作；7. 封面等增强字段。
- 主要交互：填写字段、校验必填、保存草稿、发布、取消/返回列表、未保存离开确认（P1 可增强）。
- 成功态：保存成功后给出反馈，并跳转编辑页或留在当前页展示已创建状态。
- 失败态：字段校验错误定位到字段；网络/服务失败不清空已输入内容。
- 空态：初始编辑器为空时要有明确输入引导。
- 验收标准：标题和正文等关键字段校验明确；重复 slug 或非法 slug 要有提示；保存中禁止重复提交。

### 编辑文章 `/articles/:id`（P0）

- 核心任务：编辑已有文章并管理其发布状态。
- 信息优先级：1. 当前文章标题和状态；2. 编辑字段；3. 保存/发布操作；4. 更新时间/发布时间；5. 删除或下线等危险操作。
- 主要交互：修改字段、保存、发布/下线、删除确认、返回列表、预览（P1）。
- 成功态：保存后状态和时间更新，反馈清楚。
- 失败态：文章不存在、无权限、保存失败、发布失败都要区分；失败后保留用户输入。
- 空态：文章不存在时展示不可编辑状态并提供返回列表。
- 验收标准：编辑页不能让用户误以为保存成功；危险操作有确认；离开未保存确认为 P1，但后续设计需预留。

### 分类管理 `/taxonomy/categories`（P0）

- 核心任务：维护 C 端主主题归档。
- 信息优先级：1. 分类列表；2. 名称、slug、描述、文章数量；3. 新建/编辑入口；4. 删除限制提示。
- 主要交互：新增分类、编辑名称/slug/描述、删除前确认、查看引用数量。
- 成功态：分类更新后能被文章表单和 C 端分类页使用。
- 失败态：重复 slug、被文章引用时删除失败或需明确阻止/确认；网络错误可重试。
- 空态：暂无分类时提示先创建用于组织文章主题。
- 验收标准：分类文案强调“主题归档”；不能把标签式碎片关键词放入分类。

### 标签管理 `/taxonomy/tags`（P0）

- 核心任务：维护 C 端横向检索关键词。
- 信息优先级：1. 标签列表；2. 名称、slug、文章数量；3. 新建/编辑入口；4. 合并/治理提示（后续）。
- 主要交互：新增标签、编辑名称/slug、删除前确认、查看引用数量。
- 成功态：标签更新后能被文章表单和 C 端标签页使用。
- 失败态：重复 slug、被文章引用时删除失败或需明确阻止/确认；网络错误可重试。
- 空态：暂无标签时提示标签用于框架、语言、工具、方法等横向索引。
- 验收标准：标签不做视觉标签云管理；命名治理要求避免同义词泛滥，如 `nextjs` / `next-js` / `Next.js` 混用。

### Settings 与 Media 范围裁定

- Media：本轮不作为 P0 独立页面；如当前文章表单已有封面或正文图片字段，则只要求交互不破坏流程。独立 `/media` 管理页进入 P1，范围限定为上传、选择、替换、删除确认，不做复杂素材库。
- Settings：本轮不作为 P0 独立页面；站点名称、博主信息、社交链接、SEO 基础字段进入 P1，供 Architect 和后续开发按工作量拆批。
- 验收标准：本轮 P0 不因缺少 settings/media 独立页面而失败；但导航中若出现未实现入口，必须明确禁用、隐藏或标注后续，不能跳转到空白页。

## “IT 工程师博主”内容气质验收

- 文案气质：使用具体、可信、工程化语言，优先描述“解决的问题、技术边界、实践经验、复盘结论”，避免“极致、赋能、颠覆、全网最强”等营销词。
- 视觉气质：整体克制、清晰、结构化，可有鲜明设计观点，但不得依赖紫蓝渐变、大面积装饰球、模板化卡片墙或花哨个人秀建立记忆点。
- 信息结构：首页先回答“作者是谁、写什么、从哪里读起”；列表回答“有哪些内容、如何筛选”；详情回答“如何专注读完并继续读相关内容”；Admin 回答“当前在哪、能做什么、操作结果是什么”。
- 分类与标签语言：分类应是长期主题，如工程实践、架构设计、项目复盘、工具方法；标签应是横向索引，如 Next.js、TypeScript、CI/CD、性能优化。
- 验收标准：人工审核时任取首页、文章列表、文章详情、Admin 列表和编辑页截图，应能判断这是技术内容生产与阅读场景，而不是营销站、作品集或通用后台模板。

## 人工审核点

- 定位审核：确认视觉与文案是否表达“IT 工程师博主”，而不是营销页、作品集模板或通用博客模板。
- IA 审核：依据 `information-architecture` 检查 C 端和 Admin 的导航、页面层级、内容优先级、用户流是否清晰。
- 设计审核：依据 `frontend-design` 检查是否有明确设计方向，是否克制、专业、非模板化，是否避免无意义装饰和同质化卡片堆叠。
- Web 质量审核：依据 `web-design-guidelines` 检查可访问性、可读性、布局稳定、移动端适配、交互反馈、错误/空状态。
- 截图审核：后续开发阶段必须提供 C 端桌面/移动关键页面截图，以及 Admin 登录、列表、编辑、错误、空状态等关键截图。
- 边界审核：确认未修改 `apps/`、`docs/product/`，未引入数据库迁移、生产配置、CI/CD 或真实发布动作。

## 风险

- 定位漂移：如果过度追求视觉冲击，C 端可能变成营销页或作品集，削弱技术内容可信度。
- 信息架构不足：如果先做视觉再补结构，首页、列表、详情和后台导航容易各自孤立。
- Admin 范围膨胀：后台很容易扩展到多用户、审批、统计等复杂 CMS 能力，需要坚持 P0/P1/P2 分层。
- 状态覆盖不足：若只设计理想态，登录失败、空列表、保存失败、无权限、404 等场景会破坏专业感。
- 内容依赖：C 端专业感不仅来自 UI，也依赖真实文章质量、分类命名和标签治理。
